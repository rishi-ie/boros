REFLECT] --> B[2. EVOLVE]\n    B --> C[3. VALIDATE]\n    C --> D[4. COMMIT or ROLLBACK]\n    D --> A\n```\n\n### 1. 🧠 Phase 1: REFLECT\nBoros ingests its exact operational state. It reads `logs/execution_cycles.log`, polls the `eval-bridge` for its historical benchmark scores, and analyzes the `task_log.jsonl`. If it recently suffered a regression or rollback, it pulls the post-mortem from its SQLite-backed `memory` skill. It formulates a direct hypothesis on how to overcome the capability gap preventing it from ascending to the next tier defined in its `world_model.json`.\n\n### 2. 🧬 Phase 2: EVOLVE (`meta-evolution`)\nActing upon the hypothesis, Boros executes surgical codebase mutations. It possesses the capability to:\n- Edit its own semantic `SKILL.md` instruction files (altering its prompt weights and behavioral boundaries).\n- Write, rewrite, or construct pristine Python modules inside its `functions/` directories.\n- Generate net-new operational tools via `skill-forge` representing novel capabilities.\n\n### 3. ⚖️ Phase 3: VALIDATE (`meta-evaluation`)\nBoros operates in parallel with an entirely detached, independent **Review Board Evaluator**. This separate LLM entity possesses zero context about Boros's internal evolutionary struggles. Its only job is to aggressively sandbox-test Boros's new iteration against heavily randomized, adversarial evaluation targets spawned by the `eval-generator`. It issues a comprehensive scoring composite (0.0 to 1.0).\n\n### 4. 🗄️ Phase 4: COMMIT / HOT-ROLLBACK\nIf the generated evaluation composite score equals or exceeds the absolute **High-Water Mark** registered in `state/high_water_marks.json`, the newly mutated codebase is committed permanently and hot-reloaded into memory without pausing the kernel.\nIf the score drops by even a fraction—or if the mutation triggers a catastrophic syntax crash—the entity executes a rigid safety protocol. The snapshot daemon physically reverses the `/skills` disk state referencing the `snapshots/` folder. The failure is recorded as an observation to prevent recurring bad mutations.\n\n---\n\n## 🔱 Generational Forking & Asynchronous Adaptation\n\nEvolution is not infinite straight-line progress; it requires real-world battle-testing. Boros implements a cutting-edge **Generational Paradigm**.\n\nVia the Director CLI, a human operator can execute the **`fork`** command. \n\n1. **The Freeze:** Boros immediately pauses self-mutation, finalizing the current \"Generation Epoch\" inside `lineage.json`.\n2. **The Deployment:** The entity physically changes its `agent_state` to `boros-fork`, transitioning completely from active developer into an \"employee mode.\" It handles your real-world tasks statically.\n3. **Background Adaptation Scheduler:** Boros does not sleep entirely. Under the hood, the `AdaptEngine` (`adapt_engine.py`) reads schedules established via `adapt-config` (e.g., every `2d` or `12h`). While handling field operations, it continuously analyzes user tasks, processing latent data to organically discover edge cases and operational friction.\n\nWhen the operator invokes **`re-evolve`**, Boros completes the loop. It scrapes the past field history from `task_log.jsonl`, generating a massive `adapt_seed.json` reflection matrix. The entity strips its deployment constraints, increments its generational lineage, reboots its evolution engine, and weaponizes the field data to immediately patch the exact vulnerabilities discovered during the deployment fork.\n\n---\n\n## 🌍 The World Model System\n\nYou do not write prompts.
